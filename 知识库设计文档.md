# IP 知识库设计文档

## 一、知识库是什么？

知识库本质是一个**结构化数据库**，把原来散在 Word 文档里的"哪些元素不能用、哪些组合有风险"，变成机器可以读懂的表格数据，供 AI 分析时直接调用。

---

## 二、数据库结构（3张核心表）

```
ip_profiles（IP 档案）
    ├── ip_elements（元素规则）
    └── ip_combos（组合风险规则）
```

### 表1：ip_profiles — IP 主档案

```sql
CREATE TABLE ip_profiles (
  id        INTEGER PRIMARY KEY,
  name      TEXT,           -- IP 名称，如"哈利波特"
  owner     TEXT,           -- 版权方，如"华纳兄弟"
  icon      TEXT,           -- 代表 emoji，如 🧙
  font_ref_image_path TEXT, -- 字体参考图路径（可选）
  embeddings TEXT,          -- 向量检索用的 JSON 数组（见第五节）
  created_at TEXT,
  updated_at TEXT
)
```

### 表2：ip_elements — 元素规则

```sql
CREATE TABLE ip_elements (
  id           INTEGER PRIMARY KEY,
  ip_id        INTEGER,  -- 关联 ip_profiles.id
  element_name TEXT,     -- 元素名称，如"魔法棒"
  element_type TEXT,     -- 类型（决定容忍度，见下方）
  description  TEXT      -- 审核要点说明
)
```

**element_type 五种类型及要求：**

| 类型 | 含义 | 区别度要求 |
|------|------|-----------|
| 原创首创 | 该 IP 独有，前无古人 | 需 90% 区别 |
| 原创独创 | 完全不可使用的元素 | 零容忍 |
| 非原创 | 现实中本就存在的元素 | 需 50% 区别 |
| 注册商标 | 已注册商标图形/文字 | 零容忍 |
| 原创台词 | IP 独有台词/术语 | 零容忍 |

### 表3：ip_combos — 组合风险规则

```sql
CREATE TABLE ip_combos (
  id          INTEGER PRIMARY KEY,
  ip_id       INTEGER,  -- 关联 ip_profiles.id
  rule_type   TEXT,     -- 风险类型（4种，见下方）
  description TEXT      -- 触发条件说明
)
```

**rule_type 四种组合风险类型：**

| 类型 | 说明 | 举例 |
|------|------|------|
| 外界指向性 | 组合后让人联想到该 IP | 黑袍+扫帚+猫头鹰 → 哈利波特 |
| 设计理念 | 风格理念一致导致近似 | 极简线条+特定色彩组合 |
| 产品关联性 | 该 IP 的周边产品形态 | 某IP特有的杯型/玩偶造型 |
| 现实逻辑 | 现实中不存在但IP里有的组合 | 会飞的扫帚+魔法斗篷 |

## 三、数据是怎么进来的（入库流程）

```
用户上传 .docx 文档
    ↓
mammoth.js 提取文档纯文本
    ↓
发给 Gemini AI，提示词要求提取结构化规则
    ↓
AI 返回 JSON（ipName / owner / elements[] / combos[]）
    ↓
前端展示预览，用户确认
    ↓
POST /api/knowledge/save → 写入数据库
    ↓
（异步）生成 embedding 向量，更新 ip_profiles.embeddings
```

关键：文档解析这步完全靠 AI，无需人工整理格式。

---

## 四、数据是怎么用出来的（检索流程）

```
用户提交产品描述（+ 可选图片）
    ↓
【第一步：向量检索】
    对产品描述生成 embedding 向量
    与库中所有 IP 的向量计算余弦相似度
    取相似度 > 0.7 的 Top-1 匹配
    ↓
【第二步：降级匹配（无向量时）】
    把所有 IP 名列表塞进 prompt，让 AI 猜
    ↓
【第三步：分析】
    把命中 IP 的完整规则（elements + combos）注入 prompt
    调用 Gemini 做 7 维比对
    ↓
返回风险等级 + 改款建议
```

## 五、向量检索原理

这是知识库能"语义理解"而不只是"关键词匹配"的关键。

### 什么是 Embedding（向量）？

把一段文字转换成一个高维数字数组（如 768 维），**语义相近的文字，数组也相近**。

```
"魔法棒、扫帚、霍格沃茨"  → [0.23, -0.41, 0.87, ...]
"魔法学校、巫师、魔杖"     → [0.21, -0.39, 0.85, ...]  ← 很近
"太阳能电池板、光伏组件"   → [-0.65, 0.12, -0.33, ...] ← 很远
```

### 余弦相似度

衡量两个向量"方向"是否一致，值域 [-1, 1]，越接近 1 越相似。

```
相似度 = 向量A · 向量B / (|A| × |B|)
```

本项目代码实现（`gemini.js`）：

```js
function cosineSimilarity(a, b) {
  let dot = 0, normA = 0, normB = 0
  for (let i = 0; i < a.length; i++) {
    dot += a[i] * b[i]
    normA += a[i] * a[i]
    normB += b[i] * b[i]
  }
  return dot / (Math.sqrt(normA) * Math.sqrt(normB))
}
```

### 入库时做什么

把 IP 的名称+所有元素+组合规则拼成一段话，生成 embedding 存入 `ip_profiles.embeddings`（JSON 字符串）：

```js
function buildEmbeddingText(ipName, elements, combos) {
  const elText = elements.map(e => `${e.name}（${e.type}）：${e.description}`).join('；')
  const comboText = combos.map(c => `${c.ruleType}：${c.description}`).join('；')
  return `IP名称：${ipName}。元素规则：${elText}。组合风险：${comboText}`
}
```

### 检索时做什么

对用户的产品描述同样生成 embedding，与所有 IP 向量做余弦相似度排序：

```js
const ranked = profiles
  .map(p => ({
    ...p,
    score: cosineSimilarity(queryVec, JSON.parse(p.embeddings))
  }))
  .sort((a, b) => b.score - a.score)

const topMatch = ranked[0]?.score > 0.7 ? ranked[0] : null
```

相似度阈值设为 0.7，低于这个值认为没有命中，走降级方案。

## 六、为什么不用关系型搜索（LIKE 查询）？

| 方式 | 能识别 | 不能识别 |
|------|--------|---------|
| LIKE `%哈利波特%` | "哈利波特魔杖" | "巫师学校+扫帚+猫头鹰" |
| AI 名单猜测 | IP 名出现在描述中 | 描述只有元素没有名字 |
| 向量检索 ✅ | 语义相关即可命中 | 完全无关的领域 |

向量检索的核心价值：**产品描述里不需要出现 IP 名字**，只要元素和风格语义相近就能匹配到正确的知识库。

## 七、技术栈一览

| 层 | 技术 | 用途 |
|---|------|------|
| 数据库 | SQLite（sql.js） / MySQL | 存储结构化规则和向量 |
| 文档解析 | mammoth.js | 提取 .docx 纯文本 |
| AI 解析 | Gemini（tu-zi 代理） | 文档 → 结构化 JSON |
| Embedding | text-embedding-004 | 文本 → 向量 |
| 相似度计算 | 纯 JS 实现余弦相似度 | 无需额外库 |
| 后端框架 | Express.js | REST API |
| 前端 | Vue 3 + Vite | 操作界面 |
